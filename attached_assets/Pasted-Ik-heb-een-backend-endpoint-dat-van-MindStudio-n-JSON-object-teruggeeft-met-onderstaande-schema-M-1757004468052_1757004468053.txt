Ik heb een backend-endpoint dat van MindStudio één JSON-object teruggeeft met onderstaande schema. Maak / refactor de frontend zodat dit object volledig en netjes wordt weergegeven in de pagina Mijn Zaak (React + Tailwind). Gebruik TypeScript, Zod-validatie, nette loading/error states en semantische componenten.

1) Data ophalen & valideren

Endpoint: POST /api/analyse (body bevat { input_case_details, extracted_text }; voor nu mag je met stub/testdata roepen).

Verwachte response: exact dit JSON-object (geen wrapper):

{
  "samenvatting_feiten": "string (~400 woorden)",
  "juridische_analyse": "string (~500 woorden)",
  "kwalificaties": {
    "is_kantonzaak": "string", 
    "relatieve_bevoegdheid": "string",
    "toepasselijk_recht": "string"
  },
  "vordering": {
    "hoofdsom": "string",
    "wettelijke_rente": "string"
  },
  "kansinschatting": {
    "inschatting": "kansrijk|twijfelachtig|risicovol|Onbekend",
    "redenen": ["string"]
  },
  "belangrijke_data": {
    "timeline": ["string"], 
    "deadlines_en_termijnen": ["string"]
  },
  "bewijslast": {
    "wie_moet_wat_bewijzen": ["string"],
    "beschikbaar_bewijs": ["string"],
    "ontbrekend_bewijs": ["string"]
  },
  "verjaring_en_klachttermijnen": "string",
  "conflicten_in_input": ["string"],
  "to_do": ["string"],
  "cta": ["string"],
  "kernredenering": ["string"]
}


Maak een Zod schema dat hierop matcht en valideer de response. Toon een nette fout wanneer het schema niet klopt.

Voeg een loading skeleton toe tijdens het ophalen en een retry bij error.

2) Code-locaties (pas aan als paden anders zijn)

Pagina: src/pages/MyCase.tsx (of de pagina die het “Juridische Analyse” scherm rendert).

Nieuwe/Refactor componenten in src/components/legal/:

SummaryCard.tsx – toont samenvatting_feiten.

AnalysisCard.tsx – toont juridische_analyse.

Qualifiers.tsx – badges: is_kantonzaak, relatieve_bevoegdheid, toepasselijk_recht.

ClaimPanel.tsx – kaart met vordering.hoofdsom en wettelijke_rente.

RiskPanel.tsx – statuschip o.b.v. kansinschatting.inschatting + ul met redenen.

Timeline.tsx – lijst van belangrijke_data.timeline (datum + event; onbekend → badge “Onbekend”).

Deadlines.tsx – chips uit deadlines_en_termijnen.

BurdenOfProof.tsx – drie kolommen: wie_moet_wat_bewijzen / beschikbaar_bewijs / ontbrekend_bewijs.

Limitations.tsx – paragraaf verjaring_en_klachttermijnen.

Conflicts.tsx – waarschuwing/banner wanneer conflicten_in_input.length > 0.

TodoCtas.tsx – links To-do (ul) en CTA knoppen (buttons) uit cta met onClick placeholders.

3) UI/UX regels

Typography & layout: gebruik Tailwind; headings, cards, badges, dividers.

Kansinschatting kleur:

kansrijk → groene badge,

twijfelachtig → gele badge,

risicovol → rode badge,

Onbekend → grijze badge.

Onbekend-waarden: toon ze neutraal (grijze tekst/badge) i.p.v. leeg.

Long text (samenvatting/analyse): nette leesbreedte (max-w-3xl), paragraaf-spacing.

Accessibility: aria-labels op knoppen, juiste heading-levels.

4) TypeScript modellen & Zod (maak aan in src/lib/legalTypes.ts)

Exporteer AnalysisSchema (Zod) + Analysis (TypeScript type) conform het JSON hierboven.

In de pagina: const { data, isLoading, error } = useAnalysis(caseId);

Bouw useAnalysis in src/hooks/useAnalysis.ts (fetch → validate → return typed data).

6) Acceptatiecriteria

 Data wordt opgehaald uit /api/analyse, schema-gevalideerd met Zod, en getypt in de UI.

 Elk veld uit het JSON-schema is zichtbaar (geen weggelaten secties).

 Onbekend-waarden tonen als zodanig (geen lege plekken).

 Loading skeleton en foutafhandeling met retry.

 Componenten zijn modulair en herbruikbaar.

 Geen console errors; type-safe build.
